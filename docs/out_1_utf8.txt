1. Introduction
1.1 Purpose of Phase 2
The purpose of Phase 2 is to transform the problem statement and conceptual solution identified in Phase 1 into a detailed and structured system design. This phase focuses on defining the system requirements, architectural components, and data flow necessary for implementation. Additionally, Phase 2 aims to demonstrate the feasibility of the proposed solution through a simulation-based Proof of Concept (PoC), ensuring that the design is technically viable and ready for full implementation in Phase 3.
1.2 Scope of Phase 2
The scope of Phase 2 encompasses the detailed technical design and feasibility validation of the proposed intelligent task scheduling system. Specifically, this phase includes the following components:
System design and architecture:Design of a modular and layered system architecture consisting of a web-based frontend for user interaction, a backend scheduling service responsible for task allocation, a machine learning prediction module for estimating execution time, and a relational database for persistent storage. The architecture is designed to simulate a fog-enabled distributed environment and supports containerized deployment to ensure modularity and scalability.
Detailed functional and non-functional requirements:Identification and documentation of comprehensive functional requirements covering task submission, priority handling, resource simulation, execution time prediction, heuristic-based scheduling, performance comparison, data storage, and result visualization. In addition, non-functional requirements related to system performance, reliability, usability, scalability, and security are specified to define quality constraints and operational expectations.
High-level database and data flow design:Design of a normalized relational database schema to manage task definitions, scheduling decisions, prediction outputs, and execution history. Detailed data flow descriptions are provided to explain the interaction between the frontend, scheduling logic, machine learning module, and database during task submission, prediction, scheduling, and result analysis.
Simulation-based Proof of Concept (PoC):Development of a working Proof of Concept that demonstrates the feasibility of integrating machine learning-based execution time prediction with heuristic task scheduling algorithms in a simulated environment. The PoC validates core system functionality, logical correctness, and architectural soundness without involving real-world fog hardware or production deployment.
2. System Overview
The proposed system is a simulation-based intelligent task scheduling platform designed to evaluate efficient task allocation strategies in a fog-inspired distributed computing environment. The system integrates heuristic scheduling techniques with machine learning-based execution time prediction to improve decision-making during task scheduling. Rather than relying on physical fog or cloud infrastructure, the system simulates resources and workloads to study scheduling behavior under controlled conditions.
The platform follows a modular design approach, enabling clear separation between user interaction, scheduling logic, prediction mechanisms, and data storage. This design ensures flexibility, scalability, and ease of future enhancement during Phase 3.
2.1 Product Perspective
The system is developed as a standalone academic simulation platform and does not depend on existing enterprise systems.
It operates as a web-based application, where users interact through a browser-based interface while all processing is handled by backend services.
The system architecture logically represents a fog computing environment, where computational resources are placed closer to task sources to reduce latency.
Machine learning models are used as decision-support components, assisting heuristic algorithms rather than replacing them entirely.
The system is designed to be extensible, allowing future integration with real fog or cloud environments if required.
2.2 Major System Functions
The proposed system provides a set of core functions that collectively enable intelligent task scheduling and performance evaluation in a simulated fog computing environment. Each function is designed to operate independently while contributing to the overall system workflow.
Task Definition and SubmissionThe system allows users to define computational tasks by specifying key attributes such as task size, priority level, and resource requirements. These task definitions serve as the primary input to the scheduling system and are validated before being accepted for processing.
Execution Time Prediction Using Machine LearningA dedicated machine learning module predicts the expected execution time for each submitted task. Predictions are generated using historical and simulated data, enabling the system to make informed scheduling decisions. This function enhances scheduling accuracy by reducing uncertainty associated with task execution duration.
Heuristic-Based Task SchedulingThe scheduling module applies heuristic or hybrid heuristic algorithms to assign tasks to simulated resources. Scheduling decisions are influenced by task priority, predicted execution time, and resource availability. This function aims to minimize overall execution time and improve system efficiency.
Fog Resource Simulation and ManagementThe system simulates multiple computational resources representing fog nodes with limited capacity and heterogeneous performance characteristics. Resource availability and workload distribution are dynamically managed to reflect realistic fog computing scenarios.
Scheduling Performance EvaluationThe system records scheduling outcomes such as task completion time, waiting time, and resource utilization. These metrics are used to evaluate the effectiveness of the proposed scheduling approach and to compare it with baseline scheduling strategies.
Data Logging and Result VisualizationAll task-related data, including predictions and scheduling results, are stored in a structured database. The system provides visualization and reporting features to help users analyze performance trends and system behavior.

5. System Architecture and Design
5.1 System Architecture Overview
The system follows a layered and modular architecture, consisting of the following major components:
User Interface LayerThis layer provides a web-based interface through which users interact with the system. It enables task submission, visualization of scheduling decisions, and analysis of performance metrics. The interface communicates with backend services using well-defined APIs.
Scheduling and Control LayerThis layer acts as the core decision-making component of the system. It receives task data from the user interface, invokes the machine learning prediction module, and applies heuristic or hybrid heuristic algorithms to determine optimal task-resource assignments.
Machine Learning Prediction ModuleThis module is responsible for estimating task execution time based on historical and simulated data. It operates as an independent service and provides prediction results to the scheduling module. This separation allows the prediction logic to be updated or retrained without affecting the scheduling workflow.
Resource Simulation LayerThis layer simulates fog computing resources with varying computational capacities and availability. It maintains the state of each simulated resource and provides real-time information to the scheduling module during decision-making.
Data Management LayerThe data management layer handles persistent storage of task definitions, prediction results, scheduling decisions, and execution history. A relational database structure is used to support structured queries and performance evaluation.

5.3 Data Flow Design
This section describes how data flows through the proposed intelligent task scheduling system, from initial input to processing, storage, and final output. The data flow is designed to ensure accuracy, consistency, and traceability of information across all system components in a simulation-based environment.
5.3.1 Input Sources
The system receives data from multiple input sources:
User Input:Users provide task-related information such as task size, priority, and submission parameters through the user interface. These inputs serve as the primary triggers for system processing.
System Configuration Input:Configuration parameters related to simulated fog resources, scheduling policies, and system settings are provided by the system administrator.
Historical and Simulated Data:Previously stored task execution and prediction data are used as input for machine learning-based execution time prediction.
5.3.2 Processing Stages
The processing of data occurs in a sequential and well-defined manner:
Task Validation and PreprocessingUser-submitted task data is validated to ensure completeness and correctness. Valid tasks are formatted and forwarded to the scheduling module.
Execution Time PredictionThe validated task data is passed to the machine learning prediction module, where execution time is estimated based on historical and simulated datasets.
Scheduling Decision MakingPredicted execution times, task priorities, and current resource availability are combined by the scheduling module. A heuristic or hybrid heuristic algorithm determines the optimal resource allocation.
Execution SimulationThe selected simulated resource executes the task logically, updating resource utilization and task status within the simulation environment.
Performance EvaluationSystem metrics such as task completion time, waiting time, and resource usage are computed for analysis and comparison.

5.4 Database Design
5.4.1 High-Level Database Schema
The database follows a normalized relational schema designed to represent core system entities and their relationships. The schema supports task management, execution time prediction, scheduling history, and system performance monitoring. Each entity is uniquely identifiable and linked using primary and foreign key relationships.
5.4.2 Key Entities and Relationships
The primary entities in the database include:
TaskRepresents individual computational tasks submitted by users.Each task is associated with scheduling decisions and prediction results.
ResourceRepresents simulated fog computing resources.A resource may execute multiple tasks over time but handles limited tasks concurrently.
ScheduleHistoryStores historical records of task scheduling decisions.This entity links tasks to resources and records execution timestamps.
PredictionStores machine learning-based execution time predictions for tasks.Each prediction is associated with a specific task and model version.
SystemMetricsCaptures performance-related data such as execution time, waiting time, and resource utilization.
Relationships:
A Task can have one or more Predictions.
A Task is associated with exactly one ScheduleHistory record per execution.
A Resource can be linked to multiple ScheduleHistory records.
SystemMetrics are derived from execution and scheduling data linked to tasks and resources.
5.4.3 Important Attributes
Key attributes for major entities include:
Task:TaskID, TaskSize, Priority, SubmissionTime, Status
Resource:ResourceID, ProcessingCapacity, AvailabilityStatus
ScheduleHistory:ScheduleID, TaskID, ResourceID, StartTime, EndTime
Prediction:PredictionID, TaskID, PredictedExecutionTime, ModelVersion
SystemMetrics:MetricID, TaskID, CompletionTime, WaitingTime, ResourceUtilization
These attributes enable effective tracking, analysis, and evaluation of system performance while supporting future enhancement of scheduling and prediction logic.

7.2 PoC Outcomes
Successful integration of ML prediction with scheduling logic.
Correct simulation of resource allocation and task execution.
